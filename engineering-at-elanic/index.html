6 Months of using React Native

24 January 2017

Here at Elanic, we built our iOS app using react-native (RN). When we started 7 months ago, RN was a relatively new framework and the decision to use it wasn't very clear cut. Ultimately, it came down to our need to launch an iOS app quickly and the ability for the rest of the team to contribute code. After 4 months of learning and building, we released our iOS app in November. Today, I'd like to share some of the lessons we learned using RN to develop an Indian e-commerce application.

Challenges

We already had a native android app so our primary concern was whether the react native ecosystem was mature enough to support all the external support SDKs that we use such as AppsFlyer, MixPanel, Social login (FB, Google), PayTM, etc. Luckily, there were plugins available for most of our needs except Paytm and AppsFlyer (which we decided to build ourselves).

As for performance, the standard RN constructs were performant enough for our needs. One issue was the grid view required to support variable-width views on our home page. However, By using some flexbox and ListView tricks, we were able to render a fairly performant grid view.

Confident about the RN ecosystem, we set out to develop our iOS app. It took us almost 5 months to get from idea to product and, after a lot of moving fast and breaking things, we finally launched our app in November.

Code Structure

After a fair bit of refactoring, we settled on a one-way data flow using redux as a central state for the app.

  1. The views would listen for user interactions and dispatch thunks (we're now moving business logic to sagas).
  2. The thunks would asynchronously handle network requests and dispatch the necessary actions to change state.
  3. The actions would trigger the corresponding reducers which would update the state.
    We use Immutable.js for the central state and the benefits that their persistent immutable objects give us works nicely with the react rendering logic as they allow us to track state changes across the app cheaply (by reference comparison) and update only those elements that have changed.

Automating

There are several tedious processes in app development that lend itself temptingly to automation. In our case, they included:

  1. Adding a new screen/component.
  2. Adding a new branch to the app state and associated actions/reducers.
  3. Compiling the app and deploying to beta and later promoting to production.
    We handled 1 and 2 by adding custom shell scripts to automate new file additions. 3 was handled with the help of fastlane and CodePush.

Build and deploy cycle

Fastlane allowed us to automate the deploy of new versions so we could focus on building the app. It remains an invaluable tool when we add new native features but for all non-native features, we use CodePush.

CodePush allowed us to push new features to test devices and then, to our users directly. This was invaluable in our early versions as we worked around the clock to push and deploy bug fixes to live devices. Today, CodePush is what allows us to push new features directly to devices (without waiting for an app store review). By using a suitable semantic versioning scheme, we can track the segmentation of our users and deploy accordingly.

Future

Recently, we've been working on developing a web app from the same codebase as we can re-use most of the business logic and states/reducers. The only difference would be the view logic which would be suited for the web. Using web-pack, we can selectively compile for web and iOS by naming the file *.web.js or *.ios.js. Thus differing in only the view logic and the build process, the same developers can contribute code to both web and mobile.
